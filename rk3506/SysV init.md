SysV init（System V init）是 Linux 系统中传统且经典的初始化系统和进程管理工具。它得名于 UNIX System V，是 Linux 早期广泛采用的标准。

它的核心作用是 **作为系统启动后的第一个进程（PID 1）**，负责按照严格的顺序启动、管理和维护系统中的所有其他进程和服务。

其核心工作机制基于 **运行级别**​ 和 **脚本**：

1. **运行级别**：定义了系统的一种状态（例如，0-关机，1-单用户模式，3-多用户命令行，5-带图形界面）。init 根据指定的级别来决定启动哪些服务。
    
2. **脚本**：正是您之前问到的 `/etc/init.d/`目录下的那些脚本。每个服务都有一个对应的脚本，里面定义了 `start`、`stop`、`restart`等标准操作。init 会根据运行级别，去 `/etc/rcN.d/`（N代表级别号）目录执行相应链接到 `/etc/init.d/`的脚本，从而启动或停止服务。
    

**在您的 RTU 项目中的具体关联：**

您使用的 Buildroot 默认生成的根文件系统，通常使用 **BusyBox init**，它是 SysV init 的一个轻量级实现和兼容版本。这意味着：

- 您的系统启动流程就是由它控制的。
    
- 您需要将自己编写的后台服务程序（如数据采集、Modbus/MQTT 通信守护进程）的启动脚本，按照 SysV init 的格式（支持 `start|stop|restart|status`参数）放置在 `/etc/init.d/`目录下。
    
- 然后在相应的运行级别目录（如 `/etc/rcS.d/`或 `/etc/rc5.d/`）创建以 `S`开头的软链接，系统就会在启动时自动执行它。
    

**与现代 systemd 的简单对比：**

- **SysV init**：**优点**是概念简单、脚本透明、轻量（适合嵌入式）。**缺点**是启动慢（串行执行）、依赖管理弱、脚本编写需规范。
    
- **systemd**：**优点**是启动快（并行）、依赖管理强、功能丰富。**缺点**是体系复杂、不够“UNIX哲学”。
    

对于您的嵌入式 RTU 设备，轻量、可控的 SysV init（BusyBox init）是非常典型和合适的选择。您需要掌握的就是如何为其编写规范的服务管理脚本。

# SysV init 是什么？

**SysV init = 老式 Linux 系统的 “开机总管”**

它是**传统、古老、经典**的 Linux 启动方式，名字里的：

- **SysV** = System V（一种古老的 Unix 标准）
- **init** = 系统启动后第一个运行的进程（PID=1）

---

# 它是干嘛的？

1. 系统开机后，**init 进程**第一个跑起来
2. 它按顺序启动各种服务：网络、SSH、数据库、Nginx…
3. 这些服务的启动脚本，就放在你刚才问的：
    
    plaintext
    
    ```
    /etc/init.d/
    ```
    

---

# 它的工作方式（超级好懂）

- 每个服务一个脚本：`/etc/init.d/nginx`、`/etc/init.d/sshd`
- 你用命令控制服务：
    
    bash
    
    运行
    
    ```
    /etc/init.d/nginx start
    /etc/init.d/nginx stop
    ```
    
- 开机自启靠**运行级别（runlevel）** 控制

---

# 现在还重要吗？

- **现在主流 Linux 都用 systemd 替代它了**
- 但很多老系统、嵌入式 Linux、服务器还在用
- `/etc/init.d/` 就是它留下的目录

---

# 一句话总结

**SysV init 是 Linux 传统的启动和服务管理机制，

负责开机启动各种程序，脚本放在 /etc/init.d/。

现在基本被 systemd 取代，但兼容还在。**


既然你是**学嵌入式 Linux**，那 **SysV init 就是重点**，比电脑 / 服务器上重要得多。

我直接按**嵌入式场景**给你讲透，不绕弯。

---

# 1. 嵌入式 Linux 为什么还在用 SysV init？

嵌入式特点：

- 资源少（CPU 慢、Flash 小、内存小）
- 启动流程简单、稳定
- 很多芯片原厂 BSP、SDK 还在用

**systemd 太大、太复杂，嵌入式一般不用。**

所以嵌入式里：

**init 系统 ≈ SysV init**

---

# 2. 嵌入式里 SysV init 到底干了啥？

系统上电后：

1. U-Boot 启动内核
2. 内核启动 **/sbin/init**（PID=1）
3. init 读取配置：`/etc/inittab`
4. 然后**按顺序执行启动脚本**

核心路径你必须记住：

plaintext

```
/etc/inittab
/etc/init.d/         <-- 服务脚本都在这
/etc/rcS.d/
/etc/rcX.d/
```

---

# 3. `/etc/init.d/` 在嵌入式里的真实作用

- 每个文件就是一个**服务 / 脚本**：
    
    - 挂载分区
    - 启动网络
    - 启动串口
    - 启动你的应用程序
    - 加载驱动模块
    

示例：

plaintext

```
/etc/init.d/S01mount
/etc/init.d/S50network
/etc/init.d/S99app      # 最后启动你的业务程序
```

---

# 4. 嵌入式启动顺序（你面试 / 调试必问）

1. `init` 进程启动
2. 执行 `/etc/init.d/rcS`
3. 按顺序运行 `/etc/rcS.d/` 里的脚本
    
    - S01xxx 先运行
    - S99xxx 最后运行
    
4. 最后启动登录程序或你的 APP

**这就是嵌入式 Linux 的开机启动流程。**

---

# 5. 你写嵌入式程序时会怎么用？

你以后最常用的就是：

**把自己的应用程序做成开机自启：**

1. 写一个启动脚本放到 `/etc/init.d/`
2. 软链接到 `/etc/rcS.d/S99myapp`
3. 系统开机自动运行你的 APP

---

# 极简总结（嵌入式视角）

- **SysV init = 嵌入式 Linux 的标准启动管理器**
- `/etc/init.d/ = 存放所有开机启动脚本的目录`
- 嵌入式不学这个，你连**系统怎么跑起来、程序怎么自启**都搞不懂。
