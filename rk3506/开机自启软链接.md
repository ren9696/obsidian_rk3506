我用**最直白、最贴合你嵌入式场景**的话，把这个**开机自启软链接**讲得明明白白，你一听就懂，彻底打通最后一环！

# 先给你终极结论

**开机自启软链接 = 给系统的「开机执行通知书」**

它就是一个**快捷方式**，作用只有一个：

**告诉嵌入式 Linux 开机时：自动运行你的守护脚本！**

---

# 1. 先复习两个关键目录（你必须分清）

你的系统是 **SysV init**（嵌入式标准），开机流程固定死了：

1. **/etc/init.d/**
    
    → 存放所有**服务脚本**的仓库（你的 `myapp_daemon` 在这里）
    
    → **系统开机不会主动执行这里的文件！**
2. **/etc/rcS.d/**
    
    → 系统**开机自动执行**的专属目录
    
    → 系统开机只看这个目录，按顺序执行里面的东西

---

# 2. 软链接到底是什么？

你要创建的：

plaintext

```
/etc/rcS.d/S99myapp
```

它**不是真实文件**，只是一个**指针 / 快捷方式**，

**精准指向**你的守护脚本：

plaintext

```
/etc/init.d/myapp_daemon
```

就像 Windows 桌面的「微信快捷方式」，

双击快捷方式 = 打开微信本体。

---

# 3. 为什么必须建这个软链接？

因为：

✅ 你的守护脚本在 `/etc/init.d/` → **系统开机不执行**

✅ 系统只执行 `/etc/rcS.d/` 里的文件 → **必须把脚本「放」到这里**

但你不能直接把脚本复制过去，维护麻烦，

所以用**软链接**：

**一次创建，永久生效，修改脚本不用动链接**

---

# 4. `S99myapp` 名字里的 S 和 99 是什么意思？（嵌入式必考）

这是 SysV init 的固定规则：

- **S** = Start（开机时**启动**这个服务）
- **99** = 启动优先级（数字越大，启动越晚）
    
    - S01 → 最早启动（挂载磁盘、网络）
    - S99 → **最后启动**（你的业务 APP，保证所有系统服务都准备好了）
    
- `myapp` = 随便起的名字（对应你的脚本）

---

# 5. 开机时到底发生了什么？（完整流程）

我把你三个文件的关系串起来，你瞬间通透：

1. 开发板上电
2. Linux 内核启动
3. SysV init 程序运行
4. **自动遍历 /etc/rcS.d/ 目录**
5. 找到 `S99myapp`（软链接）
6. 通过链接找到 **/etc/init.d/myapp_daemon**
7. 自动执行脚本的 `start` 命令
8. 脚本启动 **/usr/bin/myapp**（后台运行）
9. 脚本开启监控，崩溃自动重启

---

# 6. 创建软链接的命令（你直接复制用）

bash

运行

```
# 进入开机启动目录
cd /etc/rcS.d/

# 创建软链接：S99myapp → 指向 /etc/init.d/myapp_daemon
ln -s ../init.d/myapp_daemon S99myapp
```

命令解释：

- `ln -s`：创建软链接（固定写法）
- `../init.d/myapp_daemon`：链接指向的真实脚本
- `S99myapp`：开机启动清单里的名字

---

# 7. 用一个比喻彻底记住（终身不忘）

- **/usr/bin/myapp** = 灯泡（你的 APP，真正干活）
- **/etc/init.d/myapp_daemon** = 开关（控制灯泡、监控灯泡、坏了自动换新）
- **/etc/rcS.d/S99myapp 软链接** = **把开关装到墙上**
    
    → 只有装在墙上，**通电（开机）时才会自动开灯**

---

# 三者终极关系（你现在全懂了）

1. **APP**：/usr/bin/myapp → 干活的程序
2. **守护脚本**：/etc/init.d/myapp_daemon → 管理 APP、监控崩溃、自动重启
3. **软链接**：/etc/rcS.d/S99myapp → 让系统**开机自动运行脚本**
